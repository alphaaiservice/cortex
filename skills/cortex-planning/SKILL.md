# Cortex Planning — Write the Plan Before You Touch Code

The Cortex autonomous loop (`/auto-build`) only works because every phase has a written, verifiable plan in `AUTO_BUILD_STATE.json`. The same discipline applies to **any** non-trivial change — refactors, migrations, retrofits, debugging fixes, integration work. Write the plan first. Execute it second. Verify against the plan third.

This skill is the planning discipline for tasks that DON'T already have a Cortex phase plan generated by `/sprint-plan` or `/auto-build`.

---

## When to use this skill

Always fire when ALL of these are true:
- The work spans **3 or more files** OR
- The work will take **30+ minutes** of agent time OR
- The work touches **multiple Cortex layers** (controllers + services + repositories + models) OR
- The work is **hard to reverse** (migrations, dependency upgrades, schema changes, deletions)

Skip when:
- The task is a single-line fix (typo, version bump, comment) AND has zero downstream impact
- A plan already exists (`PLAN.md`, `AUTO_BUILD_STATE.json` for the current scope, sprint task with steps)
- The user explicitly says "just do it" for a clearly scoped 1-file change

When in doubt, plan. Five minutes writing a plan saves an hour of half-done implementation rollback.

## The Cortex Plan Template

Write to `PLAN.md` in the project root (or `.cortex/plans/<task-name>.md` for parallel work). Use this exact structure — downstream commands and future Cortex runs can parse it.

```markdown
# Plan — <task name>

## Goal (1 sentence)
What "done" looks like, in user-observable terms.

## Affected Cortex layers
- [ ] `app/api/v1/`       — [yes/no, which routes]
- [ ] `app/services/`     — [yes/no, which services]
- [ ] `app/repositories/` — [yes/no, which repos]
- [ ] `app/models/sql/`   — [yes/no, which entities]
- [ ] `app/models/nosql/` — [yes/no, which docs]
- [ ] `app/tasks/`        — [yes/no, which Celery tasks]
- [ ] `frontend/app/`     — [yes/no, which pages]
- [ ] `frontend/components/` — [yes/no, which components]
- [ ] `mobile/`           — [yes/no]
- [ ] `tests/`            — [yes/no, which test files]
- [ ] migrations          — [yes/no, schema impact]
- [ ] env vars / config   — [yes/no, what changes]
- [ ] docker / infra      — [yes/no]

## Files to create
1. `path/to/new_file_1.py` — [purpose]
2. `path/to/new_file_2.ts` — [purpose]
...

## Files to modify
1. `path/to/existing_1.py` — [what changes, lines approx]
2. `path/to/existing_2.ts` — [what changes, lines approx]
...

## Files to delete (if any)
1. `path/to/old.py` — [why removing]

## Dependency changes
- Add: [package@version — why]
- Remove: [package — why no longer needed]
- Upgrade: [package from→to — why]

## Build order (in this exact sequence)
1. [step 1 — depends on nothing]
2. [step 2 — depends on step 1]
3. [step 3 — can run parallel to step 2]
...

Mark parallelizable steps. Cortex personas can split work along these lines.

## Verification gates
After each step OR after the full plan, run these checks. Plan is NOT done until all pass.

- [ ] Linter clean: [exact command per language]
- [ ] Type check clean: [exact command]
- [ ] Unit tests pass: [exact command]
- [ ] Integration tests pass (if applicable)
- [ ] Manual smoke: [what to click / curl / run]
- [ ] Layer segregation intact: no api/ → repositories/ direct imports

## Risks & rollback
- Risk 1: [what could go wrong] → Mitigation: [...]
- Risk 2: [...] → Mitigation: [...]
- Rollback: [git command or migration down]

## Out of scope (for this plan)
Things explicitly NOT being done in this plan, to prevent scope creep:
1. [...]
2. [...]

## Persona ownership
- Primary persona: [Arjun | Priya | Marcus | Yuki | ...]
- Sub-tasks owned by: [persona | step #]
```

## Planning rules

### Rule 1 — One goal per plan
If the user asks for two things ("add OAuth AND switch to Postgres"), write TWO plans. Each plan has one goal. Two goals in one plan means two times the rollback complexity.

### Rule 2 — Always identify Cortex layers
The "Affected Cortex layers" checklist is non-negotiable. It surfaces hidden work (e.g., user says "add a search bar" but the plan reveals it needs a new service, a new repo, a Meilisearch dep, a docker-compose service, and a frontend component). Layer awareness is what separates Cortex planning from generic planning.

### Rule 3 — Build order matters
Sequential dependencies must be explicit. A subagent that tries to write the controller before the service will hallucinate the service interface. The build order is the API the personas use to coordinate.

### Rule 4 — Verification gates are part of the plan, not after
List the lint/type/test commands BEFORE writing code. This forces you to know what success looks like before you commit to a path. If you can't name the verification command upfront, the plan is incomplete.

### Rule 5 — Out-of-scope list is mandatory
Every plan must declare 2+ things it is NOT doing. This prevents the "while I'm in here..." scope explosions that destroy timelines.

## When to update an existing plan

Replace, don't append:
- If the goal changes → write a new plan, archive the old one to `.cortex/plans/archive/`
- If the build order is wrong → rewrite the build-order section, don't add "actually..." notes
- If verification fails → update the plan with what went wrong + new mitigation, then re-execute

Append:
- New files discovered as needed (add to "Files to create/modify")
- New risks discovered (add to "Risks & rollback")
- Steps completed (cross them out with `~~strikethrough~~` and add a "DONE" tag)

## Anti-patterns

- ❌ **Verbal plans only.** "I'll just edit these few files" — no. Write it down. Five minutes.
- ❌ **Plans without verification gates.** A plan without "how do I know it worked" is a wish.
- ❌ **Skipping the "Affected layers" checklist.** It exists to surface hidden work.
- ❌ **Mixing goals in one plan.** Each goal = one plan.
- ❌ **Plans without "out of scope".** Scope creep is the default; explicit anti-scope is the cure.
- ❌ **Treating the plan as immutable.** Plans are working documents. Update them when reality differs.
- ❌ **Skipping plans for "small" changes that touch 3+ files.** Three files is the threshold. Trust it.

## Integration with other Cortex skills

- **`cortex-brainstorming`** — runs BEFORE this skill when the task is ambiguous. Brainstorming produces intent; this skill produces the implementation plan.
- **`cortex-tdd`** — the "Verification gates" section maps directly to TDD's red→green→refactor loop.
- **`cortex-verification`** — runs the verification gates declared in the plan, gates `/ship` and PR-merge.
- **`/sprint-plan`** — when the work is product-scale (multiple features, weeks of effort), use `/sprint-plan` instead. This skill is for single-task multi-file work.
- **`/auto-build`** — the plan format here is intentionally compatible with `AUTO_BUILD_STATE.json` phase entries — auto-build can ingest a `PLAN.md` as a phase if you mark it `phase_compatible: true` in the frontmatter.
